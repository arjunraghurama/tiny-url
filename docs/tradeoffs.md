# ⚖️ Design Trade-offs

Every system design decision involves trade-offs. Here are the key decisions we made and their alternatives.

## 1. Short Code Generation: Random vs Sequential

### Our Choice: Random Alphanumeric Codes

```python
# Cryptographically random code
secrets.choice("a-zA-Z0-9") × 7 → "kX9mBzQ"
```

| | Random Alphanumeric | Base62 from ID |
|---|---|---|
| **Collisions** | ❌ Possible (need retry) | ✅ Impossible |
| **Predictability** | ✅ Unpredictable | ❌ Sequential — enumerable ⚠️ |
| **Complexity** | ✅ Simple | ✅ Simple |
| **Custom codes** | ✅ Independent of ID | ❌ Tied to DB sequence |
| **Security** | ✅ No enumeration | ❌ Attacker can scan all URLs |
| **Performance** | ✅ O(1) | ✅ O(1) |

!!! question "When would you choose Base62 from ID instead?"
    If you're building an **internal-only** tool where enumeration isn't a concern and you want to guarantee zero collisions without any retry logic.

## 2. Database: SQL vs NoSQL

### Our Choice: PostgreSQL (SQL)

| | PostgreSQL (SQL) | DynamoDB/Cassandra (NoSQL) |
|---|---|---|
| **ACID** | ✅ Full transactions | ❌ Eventual consistency |
| **Schema** | ✅ Enforced | ❌ Schemaless |
| **Joins** | ✅ Efficient | ❌ Not supported |
| **Scaling** | ❌ Vertical + Read replicas | ✅ Horizontal by default |
| **Complexity** | ✅ Well-understood | ❌ Different mental model |
| **Analytics** | ✅ Complex queries | ❌ Limited query patterns |

!!! question "When would you choose NoSQL?"
    At **massive scale** (billions of URLs) where horizontal scaling is critical and you don't need complex queries. Companies like Bitly use a mix of both.

## 3. Caching: Cache-Aside vs Write-Through

### Our Choice: Cache-Aside (Lazy Loading)

| | Cache-Aside | Write-Through | Write-Behind |
|---|---|---|---|
| **Read latency** | ❌ First read is slow | ✅ Always cached | ✅ Always cached |
| **Write latency** | ✅ Fast writes | ❌ Slower writes | ✅ Fast writes |
| **Memory usage** | ✅ Only hot data | ❌ All data cached | ❌ All data cached |
| **Consistency** | ❌ TTL-based staleness | ✅ Always fresh | ❌ Async, may lose data |
| **Complexity** | ✅ Simple | ✅ Medium | ❌ Complex |
| **Cache failure** | ✅ System still works | ❌ Writes fail | ❌ Data loss risk |

!!! question "When would you choose write-through?"
    If you need **strong consistency** and every URL must be immediately available in cache after creation. The trade-off is higher write latency and more memory usage.

## 4. Redirect: 301 vs 302 vs 307

### Our Choice: 307 Temporary Redirect

| Code | Type | Browser Caching | Click Tracking | Method Preserved |
|---|---|---|---|---|
| 301 | Permanent | ✅ Cached forever | ❌ Loses clicks | ❌ May change to GET |
| 302 | Found | ❌ Not cached | ✅ Trackable | ❌ May change to GET |
| **307** ✅ | Temporary | ❌ Not cached | ✅ Trackable | ✅ Preserved |

!!! question "When would you choose 301?"
    If you **don't need click tracking** and want to reduce server load. SEO tools like URL canonicalization often use 301s.

## 5. ID Type: Auto-Increment vs UUID vs Snowflake

### Our Choice: Auto-Increment BigInt (for URL ID)

| | Auto-Increment | UUID v4 | Snowflake ID |
|---|---|---|---|
| **Size** | 8 bytes | 16 bytes | 8 bytes |
| **Sortable** | ✅ Natural order | ❌ Random | ✅ Time-sorted |
| **Distributed** | ❌ Single source | ✅ Any node | ✅ Any node |
| **Predictable** | ❌ Yes | ✅ No | ❌ Partially |

!!! note "For Users, we use UUID"
    The `users` table uses `UUID` as primary key because user IDs may be generated by external auth systems (Keycloak, Auth0) and need to be globally unique without coordination.

## 6. Async vs Sync Processing

### Our Choice: Async FastAPI + asyncpg

| | Async (Our choice) | Sync (Flask/Django) |
|---|---|---|
| **Concurrent connections** | ✅ Thousands | ❌ Thread-limited |
| **I/O bound workloads** | ✅ Excellent | ❌ Wastes threads waiting |
| **Complexity** | ❌ async/await everywhere | ✅ Simple |
| **Debugging** | ❌ Stack traces harder | ✅ Straightforward |
| **Ecosystem** | ❌ Need async libraries | ✅ More library choices |

!!! tip "URL shorteners are I/O bound"
    Every request does network I/O (cache lookup + DB query). Async is the perfect fit because we're waiting on I/O, not computing heavy calculations.

## Summary Decision Matrix

| Decision | We Chose | Because |
|---|---|---|
| Short code generation | Random alphanumeric | Unpredictable, no enumeration |
| Database | PostgreSQL | ACID, mature, great for learning |
| Cache pattern | Cache-aside | Simple, fault-tolerant, read-optimized |
| Redirect type | 307 Temporary | Track clicks, preserve methods |
| URL ID type | Auto-increment | Internal use only, efficient storage |
| User ID type | UUID | Future auth integration |
| Framework | Async FastAPI | I/O bound workload, high concurrency |
| Cache store | Valkey | Redis-compatible, open source |
